## Why this is a HOLD, not a build

The only instruction that reached this session was the four-word gloss "export-time per-side bleed normalization." That phrase is compatible with two genuinely different features (see "Two candidate designs" below) with very different risk profiles — one is a pure UI/preview addition, the other is an image-heuristic that can silently mis-crop a paying customer's cards if the heuristic is wrong. Guessing which one and building it blind isn't a reasonable substitute for the actual spec. This doc surveys the current architecture, proposes both candidates concretely enough to choose between, and recommends one — but the choice is flagged as an open decision, not assumed.

## Survey: how bleed is handled today

`frontend/src/features/pdf/PDF.tsx`'s `PDFCardImage` (current master, post-Proposal-A) renders every card slot at `CardWidthMM + 2*bleedEdgeMM` × `CardHeightMM + 2*bleedEdgeMM`, where `bleedEdgeMM` is a single **global, user-configurable** value from the PDF settings panel (`PDFProps.bleedEdgeMM`, defaulting to `BleedEdgeMM` = 3.048mm, `frontend/src/common/constants.ts`). The source image itself is assumed to already be bled at exactly the standard `BleedEdgeMM` amount on all four sides — the component computes one `widthProportion`/`heightProportion` scale factor (`(CardWidthMM + 2*BleedEdgeMM) / (CardWidthMM + 2*bleedEdgeMM)`, same formula for height) and applies it as a single uniform `transform: scale(...)` to the whole image. This is **not per-side** today, and it's **not a correctness check** — it's a uniform re-scale from "the standard bleed amount" to "whatever bleed amount the user asked for," with no awareness of whether the source image's real bleed matches the standard amount at all, let alone whether it matches evenly on all four edges.

Two things this can't handle, both real:
1. **A source image with no real bleed at all** (a trimmed scan) gets scaled exactly like a properly-bled one — the result is a printed card whose art/border sits in the wrong place relative to the cut line, or bleeds white at the trim line, depending on print-shop tolerance.
2. **A source image with real but asymmetric bleed** (e.g. a slightly off-center scan — more margin on the left than the right) gets one X-scale and one Y-scale applied uniformly — this can't correct an asymmetry that's already baked into the pixels.

The backend already solved a version of problem 1, but not client-side and not per-edge: `docs/features/printing-tags.md`'s "Bleed-edge tagging" section (`local_fallback.py`'s `classify_bleed_edge`) classifies a fetched image as `'bleed'` or `'trimmed'` by comparing its aspect ratio against two reference constants — `TRIM_ASPECT_RATIO = 63/88 ≈ 0.7159` vs. `BLEED_ASPECT_RATIO = 69.35/94.35 ≈ 0.7350` (both derived from `CardWidthMM`/`CardHeightMM`, exactly the constants this frontend already imports) — abstaining when the ratio is more than 0.03 from both. Validated against a real 40-source sample: clean bimodal separation, ~97.5% `'bleed'`, one clear `'trimmed'` outlier, nothing observed in the gap. That's a genuinely reusable, already-proven technique — but it's a **whole-image binary classification**, not a per-edge measurement, and the resulting `appropriate-bleed` tag currently has **0 cards resolved** in production (it only casts a negative vote for the rare `'trimmed'` case, and that vote still needs a moderator co-sign to land in `card.tags` before the frontend could ever read it — see the tag's `SENSITIVE` classification). Any frontend design that depends on reading this tag from the API today would have essentially no data to act on.

## Two candidate designs

### Design 1 — automatic per-edge detection + auto-correct

At PDF render time (or at image-load time, cached per identifier), analyze each of the four edges of the loaded image independently — e.g. sample a thin band along each edge and look for the kind of uniform, low-variance color run real bleed margins usually have, vs. the higher-variance transition where card content abuts a hard trim line. Where an edge looks short on real margin, synthesize the missing bleed by edge-extending (repeating/mirroring the outermost row or column of pixels) just that side, independently of the other three, via an offscreen `<canvas>` before handing the result to `@react-pdf/renderer`'s `<Image>`.

- **Pro**: zero user interaction — "just works" for the common case, including the ~2.5% `'trimmed'` real-world minority the backend's own sample found.
- **Con**: it's a heuristic operating on community-sourced images of wildly varying quality (scan noise, JPEG artifacting, watermarks near edges, borderless full-art prints whose actual card content legitimately extends near the edge) — a false positive silently crops or distorts a real customer's print order, and this session has no way to validate a border-detection heuristic against a representative image sample the way the backend's bleed classifier was (40-source, ground-truth-checked, abstain threshold). Building this blind risks shipping something confidently wrong.
- **Effort**: medium-large. New pure module (`bleedNormalize.ts`, testable independent of `<canvas>` the way `layout.ts` is independent of `@react-pdf/renderer`), an actual edge-detection algorithm needing real-image validation before trusting it (the backend's own bleed-edge work took a dedicated validation pass before shipping even the simpler binary classification — this is strictly harder, four independent measurements instead of one), a `<canvas>`-based synthesis step wired into `PDFCardImage`'s image-loading path, and a way to surface "which cards got auto-corrected" so it isn't silent (mirrors this session's own "no silent caps" instinct elsewhere).

### Design 2 — manual per-side controls in the existing WYSIWYG preview (recommended MVP)

Proposal A already built a WYSIWYG CSS-transform page preview (`PagePreview.tsx`) that renders the actual computed layout live, card-by-card, before the user commits to a PDF render. Design 2 adds a **per-card, per-side bleed offset** the user can nudge (four small +/- controls or a drag-handle per edge, scoped to mm, reasonable range e.g. ±3mm) directly against that live preview — visible immediately since the preview already re-renders on any layout input change. The offsets are pure numbers threaded through to `PDFCardImage`'s existing transform math: instead of one uniform `scale()`, each edge gets its own independent crop/extend margin in the final slot.

- **Pro**: no heuristic to get wrong — the user's own eyes are the detector, using a preview surface that already exists. Zero risk of a silent bad auto-correction. Buildable incrementally: start with a single "override this card's bleed" affordance triggered from the preview (not a permanent always-visible control on every card), so the common case (properly-bled source images, the ~97.5%) stays untouched and uncluttered.
- **Con**: doesn't help someone who never opens the preview or doesn't notice an issue there; scales awkwardly to a large project (per-card controls across hundreds of cards) unless scoped carefully (see open decisions below).
- **Effort**: small-medium. `layout.ts`'s `computeLayout()`/`LayoutSlot` already carries per-slot page-absolute placement — extending it with an optional per-slot bleed-offset override (defaulting to today's uniform behavior when absent) is additive, not a rewrite. `PagePreview.tsx` gets the interactive controls; `PDF.tsx`'s `PDFCardImage` reads the same override when actually rendering, so preview and output stay in sync by construction (the same reason `layout.ts` was extracted in the first place — one source of truth, not two independently-tuned code paths).

### A middle path worth naming, not fully speccing

Design 1's detection half (per-edge measurement) could feed Design 2's manual controls as a **suggestion**, not an auto-apply — "this card's right edge looks short on bleed, tap to compensate" — reusing this session's own established suggestion-vs-confirmed vocabulary (the funnel's chip-lean tooltips, the audit-build-pass PR #64) instead of silently correcting. This gets Design 1's "catches it for you" value without Design 1's silent-wrong-crop risk, at the cost of needing both halves built. Flagged as a plausible v2, not proposed for the initial build.

## Recommendation

Build Design 2 first. It's smaller, carries no heuristic-correctness risk, and reuses Proposal A's preview surface rather than adding a new one. Design 1 (or the middle path) stays a real option for a later pass, once there's a way to validate an edge-detection heuristic against a real image sample the way the backend's own bleed-edge classifier was validated — that validation step isn't optional groundwork to skip, per this session's own precedent (E-2, the Level 2 fix, and the backend's bleed-edge section above all did a real-data check before trusting a heuristic).

## Open decisions (owner sign-off needed before code)

1. **Design 1 vs. Design 2 vs. the middle path** — the core choice above.
2. **Scope of manual controls, if Design 2**: per-card override triggered from the preview (recommended) vs. a project-wide set of four sliders applied to every card uniformly (simpler UI, but doesn't solve the asymmetric-per-image problem Design 2 exists to solve) vs. both (global default + per-card override).
3. **Should the frontend read `appropriate-bleed` from `card.tags` at all**, given it's currently unpopulated (0 resolved) — worth wiring as a future-proofing hook (a already-`'trimmed'`-tagged card pre-fills a suggested correction) even though it'd be a no-op today, or skip it until the backend signal actually has data.
4. **Persistence**: per-card bleed overrides live in Redux project state (persisted like everything else in `projectSlice`) or are export-session-only (reset on reload)? Precedent both ways exist in this codebase (the Local Folder directory handle is deliberately session-only; most project state persists).

## Effort estimate

Design 2 (recommended): ~1-2 focused sessions — `layout.ts` extension + tests (small), `PagePreview.tsx` per-side controls + tests (medium, new interactive UI), `PDF.tsx` wiring to consume the same override (small, mirrors existing `bleedEdgeMM` plumbing). Design 1: meaningfully larger — a real validation pass before any code ships, per the recommendation above, plus the detection/synthesis implementation itself.
