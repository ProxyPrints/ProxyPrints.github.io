is therefore the pilot-era snapshot used throughout — inferred from commit

checkable rather than assumed.

### Legacy-engine-only "fetch cards below a resolved-negative tag should never be selected" — resume/rescan semantics

Stage C's resume filter (`run_image_evidence_cohort.py`) marks a card "already fully processed" once its `ImageEvidence.extractor_versions` contains every manifest key — which every extractor sets **even when it degrades to a skip** (documented explicitly: "Set even on skip — fetch_health RAN TO COMPLETION either way"). This means a transient fetch failure is treated identically to a genuine successful extraction for resume purposes, unlike the pilot's `RESCANNABLE_SKIP_REASONS` distinction between transient (`unfetchable-image`) and permanent skips. The replacement mechanism is explicit, not automatic: `--card-ids-file` / `reparse_collector_evidence`'s `--selector` flags force a targeted re-run. Classified as **moved from automatic to explicit re-scan**, not silently dropped — the capability exists, just requires an operator decision instead of firing on the next scheduled invocation.

## MISSING — stated plainly

Three items, confirmed absent by direct `grep`/read of `local_calculate_verdicts.py`, `image_evidence.py`, and `run_image_evidence_cohort.py` — not assumed from the design docs' own claims:

1. **`RESOLUTION_FLOOR_DPI = 200`** (pilot: `local_identify_printing_tags.py`). The pilot's own 6-way dpi sweep found dpi≤150 measurably degrades OCR yield and excluded any card whose source image sits below this floor from being fetched _at all_. Stage C's cohort selection (`run_image_evidence_cohort.py`) has no `dpi` condition anywhere in its eligibility query — it fetches every `content_phash`-populated card regardless of source resolution. This is the most safety/yield-relevant of the three: a genuinely low-resolution source image now gets a full OCR/phash/legal-line extraction pass that the pilot's own measured data said would underperform, with no signal anywhere downstream (`ImageEvidence` carries no dpi/resolution field) to distinguish those cards later.

2. **`EXCLUDED_RESOLVED_TAGS = ["custom-art", "non-english"]`** (pilot). A card already carrying a resolved custom-art or non-english tag has its printing-identification precondition (an authentic depiction of a real printing) already falsified — the pilot excluded these from selection entirely. Stage D's `_eligible_cards_queryset` in `local_calculate_verdicts.py` has no equivalent `.exclude(tags__contains=...)` clause. Confirmed by `grep` for `custom-art`/`non-english`/`EXCLUDED_RESOLVED_TAGS`/`tags__contains` across all three Stage C/D files: zero hits. Checked for a same-effect-different-mechanism explanation before calling this MISSING (not just absence-of-the-literal-clause): `tag_consensus.py`'s `resolve_and_persist_tag_votes` only ever writes `card.tags`/`card.tag_vote_statuses` — it never touches `printing_tag_status` or `canonical_card`, so a resolved custom-art/non-english tag does NOT get a card excluded through Stage D's `printing_tag_status=UNRESOLVED`/`canonical_card__isnull=True` filters either; there is no other path that produces the pilot's exclusion effect. This does not create a soundness violation (the human-backed consensus gate still applies to any vote cast), but it does mean Stage D would spend real OCR/phash effort — and potentially cast a real `is_no_match` or match vote — on a card whose own tags already say the exercise is moot, which the pilot's own design explicitly avoided. Not previously tracked anywhere found in `docs/features/catalog-completion-plan.md`/`docs/theory.md` — appears to be a genuinely new finding from this sweep, not a known, already-accepted gap.

3. **`.exclude(printing_tags__anonymous_id=DEDUCTIVE_BACKFILL_ANONYMOUS_ID)`** (pilot). Lower severity than the two above, and narrower than it first looks: verified against `printing_consensus.py` directly — a card deductive backfill has actually _resolved_ (`resolve_and_persist_printing` sets `inferred_canonical_card` + `printing_tag_status=RESOLVED`) is already excluded from Stage D's query by its own pre-existing `printing_tag_status=UNRESOLVED` filter, no separate exclusion needed. This gap therefore only bites the narrower "deductive backfill already cast a vote for this card, but it's still sitting UNRESOLVED (pending human confirmation, or the vote alone wasn't sufficient)" subset — the pilot deliberately didn't let its own weaker engines pile a redundant vote onto exactly that subset. Since Stage D's own vote is still gated by the same human-backed consensus rule, this is a redundancy/efficiency gap on that narrower subset, not a soundness one — included for completeness, not alarm.

No other pilot-era constant, threshold, crop box, regex, or skip-reason was found with genuinely no current home; every other divergence found a place in one of the non-MISSING buckets above, including several where the current home is the SAME code, literally imported and called (not reimplemented) by the new pipeline.
